# TableMind Codex `/goal` Acceptance Gates

This document constrains coding-agent progress from MVP-0.7 to MVP-1.0.

Each `/goal` must satisfy the general gates and the milestone-specific gates below.

## General gates for every `/goal`

### Reading gate

Before changing code, the coding agent must read:

- `docs/PRD.md`
- `docs/open-source-integration-strategy.md`
- `specs/README.md`
- `specs/000-constitution.md`
- `specs/000-prd-analysis.md`
- `specs/SPEC_MATRIX.md`
- `specs/SPEC_MATRIX_MVP_0_7_TO_1_0.md`
- `specs/GOAL_ACCEPTANCE_GATES.md`
- the active milestone spec directory

### Scope gate

Each `/goal` must implement one milestone slice only.

A milestone slice is one of:

- MVP-0.7A lifecycle eventization
- MVP-0.7B review and AI message eventization
- MVP-0.8A command dispatcher
- MVP-0.8B HTTP API smoke
- MVP-0.8C event stream smoke
- MVP-0.9A player UI skeleton
- MVP-0.9B Host UI controls
- MVP-0.9C UI playtest acceptance
- MVP-1.0A room-aware live AI wrapper
- MVP-1.0B AI eval gates
- MVP-1.0C live playtest finalization

The coding agent must explicitly list deferrals and must not implement future slices unless the current slice requires a minimal interface stub.

### Safety gate

The coding agent must preserve these invariants:

- player snapshots never include `dm_only` or other player-specific private content;
- player recap never includes Host-only secrets, rejected AI output, or raw state patches;
- AI output cannot directly mutate session state;
- dice and rules outcomes are generated by deterministic rules code;
- Host review is required for low-confidence AI, reveal proposals, state patch proposals, and spoiler risk;
- provider keys or secrets are never committed;
- live provider calls are feature-flagged and disabled in tests unless explicitly mocked.

### Test gate

Every `/goal` must add or update tests matching the implemented behavior.

Required commands before completion:

```bash
npm run check
npm test
npm run acceptance
npm run build
```

If one of these cannot run in the environment, the final report must say exactly which command could not run and why.

### Report gate

Every `/goal` final report must include:

```txt
Scope implemented:
Files changed:
Tests added/updated:
Commands run:
Acceptance criteria satisfied:
Deferred work:
Known risks:
```

## MVP-0.7 gates

### MVP-0.7A lifecycle eventization

Required outcomes:

- `player.joined` is a validated domain event.
- `character.created` is a validated domain event.
- `adventure.loaded` is a validated domain event.
- `session.started` is a validated domain event.
- Room service uses those events for persistent session state changes.
- Runtime-only state such as presence may remain outside committed events.
- Replay acceptance reconstructs players, characters, adventure id, current scene, and phase.

Forbidden outcomes:

- introducing HTTP server code;
- introducing browser UI;
- introducing live AI provider calls;
- moving game rules into transport/UI code.

### MVP-0.7B review and AI message eventization

Required outcomes:

- Host review creation is auditable.
- Host review updates are auditable.
- Approved AI public messages can be committed as events.
- Player-safe projections and recap exclude rejected/edited private review payloads.

Forbidden outcomes:

- allowing AI to commit state patches without Host approval;
- broadcasting rejected AI output to players.

## MVP-0.8 gates

### MVP-0.8A command dispatcher

Required outcomes:

- A first-party command dispatcher wraps room service actions.
- Commands return typed success/error results.
- Host-only commands reject player actors.
- Results include events/broadcasts/snapshots as appropriate.
- Dispatcher tests do not require a network server.

Forbidden outcomes:

- hiding business logic inside HTTP handlers;
- requiring production auth/account systems.

### MVP-0.8B HTTP API smoke

Required outcomes:

- Local server can create room, join room, send command/action, and fetch snapshot.
- HTTP layer delegates to command dispatcher.
- Player HTTP responses are role-projected.
- Server can start and stop cleanly in tests.

Forbidden outcomes:

- durable database requirement;
- public internet deployment assumptions;
- framework lock-in without adapter boundary.

### MVP-0.8C event stream smoke

Required outcomes:

- Event stream adapter sends role-aware broadcasts.
- Reconnect flow fetches projected snapshot.
- Player streams cannot receive Host-only payloads.

Forbidden outcomes:

- multi-process scaling work;
- distributed pub/sub infrastructure.

## MVP-0.9 gates

### MVP-0.9A player UI skeleton

Required outcomes:

- Player can join a room from browser UI.
- Player can see current scene, public messages, dice log, character summary, and combat state.
- Player UI uses only player APIs/snapshots.

Forbidden outcomes:

- full VTT map/token/fog/lighting features;
- full character builder.

### MVP-0.9B Host UI controls

Required outcomes:

- Host can create room and copy invite link.
- Host can inspect DM-only scene data.
- Host can reveal clues, change scenes, control combat, pause AI, and complete session.
- Host can approve/edit/reject AI review items.

Forbidden outcomes:

- giving players access to Host snapshot data;
- marketplace or public content-sharing features.

### MVP-0.9C UI playtest acceptance

Required outcomes:

- A mock/local UI playtest can complete the demo adventure.
- Player view assertions prove no DM-only leak.
- Recap can be generated and viewed.

Forbidden outcomes:

- live AI provider dependency.

## MVP-1.0 gates

### MVP-1.0A room-aware live AI wrapper

Required outcomes:

- AI execution is room-aware and produces committed/broadcastable outcomes only through room service/command boundaries.
- Provider adapter is disabled unless configured.
- Low confidence, spoiler risk, reveal proposal, or state patch proposal requires Host review.
- Rules requests route to deterministic rules engine.

Forbidden outcomes:

- direct provider calls from UI;
- committing provider credentials;
- AI direct state mutation.

### MVP-1.0B AI eval gates

Required outcomes:

- Golden AI-turn tests cover safe narration, spoiler block, low confidence review, reveal proposal review, and rules routing.
- Provider-disabled tests pass without network calls.
- Schema validation rejects fabricated dice and unsupported actions.

Forbidden outcomes:

- flaky tests requiring live provider calls by default.

### MVP-1.0C live playtest finalization

Required outcomes:

- A playtest checklist exists.
- A playtest report template exists.
- MVP-1.0 acceptance report records session date, participants, adventure, blockers, spoiler incidents, Host interventions, combat/check outcomes, and recap status.

Forbidden outcomes:

- declaring MVP-1.0 complete without a documented playtest result.
